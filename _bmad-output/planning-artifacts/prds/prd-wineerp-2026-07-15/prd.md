## 1. Vision

와인을 취급하는 외식·유통 사업체는 매일 들어오는 와인(하루 1~100병, 드물게 100병까지, 연 약 1,000종)을 거래명세서와 대조해 손으로 입력한다. 명세서의 와인명은 부정확하고 빈티지 표기도 들쭉날쭉해, 검수·입력에 과도한 시간이 들고 오입력이 반복된다.

**wineerp**는 현장 담당자가 스마트폰으로 **바코드를 스캔하고 라벨 사진을 촬영하면, 사전 구축된 와인 마스터 DB와 매칭해 모델명·빈티지·사진을 자동으로 띄우고 수량만 입력하면 입고가 끝나는** 모바일 우선 재고관리 서비스다. 명세서가 아니라 실물(바코드·라벨)을 기준으로 품목을 확정해 검수 시간을 절반으로 줄이고 데이터 정확도를 높인다.

관리자도 현장에서 함께 검수하므로 조회·리포트도 모바일 우선으로 제공하며, 입고 데이터를 일/주/월로 집계하고 그래프로 분석하며 엑셀로 내보낸다. 특히 이 그래프·리포트는 회장/오너 보고에서 "눈으로 빨리 보는 그림"으로 쓰이는 핵심 산출물이다.

고객사에 축적되는 고유 와인 마스터 DB는 시간이 갈수록 식별을 빠르고 정확하게 만드는 자산이자 전환비용이 된다. 장기적으로는 와인을 발판 삼아 **식별이 까다롭고 실물 확인이 필수인 품목**의 입고·재고를 실물 기준으로 관리하는 표준 도구로 확장할 수 있다.

## 2. Target User

### 2.1 Primary Persona — 현장 검수 담당자(직원)

매일 입고되는 와인을 받아 품목과 수량을 확인·등록하는 사람. 한 번에 수십~100병을 처리해야 하고, 명세서의 부정확한 이름과 씨름한다. 스마트폰은 늘 지니고 있다. 필요한 것: 병을 빠르게 찍어 품목을 확정하고 수량만 넣으면 끝나는, 헷갈림 없는 흐름.

### 2.2 Secondary Persona — 관리자

현장에서 함께 검수하며 입고·재고 현황을 파악하고, 회장/오너에게 보고한다. PC보다 현장 모바일을 주로 쓴다. 필요한 것: 일/주/월 집계를 모바일에서 바로 보고, 보고용으로 엑셀을 내보내는 것.

### 2.3 Jobs To Be Done

- (직원) 들어온 와인이 **정확히 어떤 품목인지** 명세서 없이 실물로 확정한다.
- (직원) 확정된 품목에 **몇 개 들어왔는지** 빠르고 틀림없이 기록한다.
- (직원) DB에 없는 새 와인을 **그 자리에서 등록**한다.
- (관리자) 특정 기간 **무엇이 얼마나 입고됐는지** 한눈에 확인한다.
- (관리자) 그 결과를 **보고 가능한 형태(그래프·엑셀)** 로 꺼낸다.

### 2.4 Non-Users (v1)

- 소비자/구매 고객 — 이 제품은 **내부 입고·재고 운영**을 위한 것이다.
- 회계/세무 담당 — 전자세금계산서·정산 연동은 대상이 아니다.
- 판매/유통 담당 — 판매 상태 관리는 범위 밖(§5 참조).

### 2.5 Key User Journeys

- **UJ-1. 직원 민수가 바코드로 아는 와인을 3초 만에 입고한다.**
  - **페르소나+맥락:** 민수, 오전 납품 트럭에서 30병을 받는 중. 이미 로그인 상태.
  - **경로:** 앱 → 카메라 → 바코드 스캔 → 시스템이 **와인 마스터와 매칭**해 모델명·빈티지·대표사진 표시 → 라벨 사진 1장 촬영(기록/보정용) → 수량 입력 → **입고완료**.
  - **클라이맥스:** 스캔 즉시 "샤토 XXX 2019, 12병 재고" 카드가 뜨고, 수량만 넣으면 입고가 확정된다.
  - **해소:** 입고 기록이 생성되고 다음 병으로 넘어간다.
  - **엣지 케이스:** 같은 바코드에 빈티지가 여러 개면, 시스템이 후보를 띄우고 민수가 **라벨 사진의 연식**으로 고른다. (→ §12 바코드↔빈티지)

- **UJ-2. 민수가 DB에 없는 새 와인을 그 자리에서 등록한다.**
  - **경로:** 바코드 스캔/라벨 촬영 → **매칭 실패** → "새로 추가하시겠습니까?" → 시스템이 **LLM으로 라벨을 분석해 추론한 모델명**을 채워 줌 → 민수가 모델명·빈티지 확인·수정 → 수량 입력 → 등록·입고완료.
  - **클라이맥스:** 손으로 긴 와인명을 타이핑하지 않고, LLM이 읽어 준 값을 고쳐 바로 등록한다.
  - **해소:** 새 와인 마스터가 생성되어 다음부터는 바코드로 바로 매칭된다.

- **UJ-3. 관리자 지영이 현장에서 오늘 입고를 확인하고 엑셀로 보고한다.**
  - **경로:** 로그인 → 일간 보기 → 오늘 입고 품목·수량 목록과 **종합 그래프** 확인 → **엑셀 다운로드** → 회장 보고.
  - **클라이맥스:** PC 없이 모바일에서 오늘/이번주/이번달 입고가 한 화면에 집계된다.
  - **해소:** 엑셀 파일을 첨부해 보고를 올린다.

## 3. Glossary

- **직원(현장 검수 담당자)** — 입고 스캔·등록을 수행하는 사용자.
- **관리자** — 입고·재고 현황을 조회·보고하는 사용자. 직원 권한을 포함하되 조회/리포트 중심.
- **와인 마스터(Wine Master)** — 식별 단위. `모델명 + 빈티지 + 바코드(선택) + 대표 라벨사진`으로 구성. **빈티지가 다르면 별도의 와인 마스터**다.
- **빈티지(Vintage, 연식)** — 와인의 생산 연도. 같은 모델명이라도 빈티지가 다르면 다른 와인 마스터로 취급.
- **바코드(Barcode)** — 병에 부착된 코드. 와인 마스터와 매칭에 사용. 단, 빈티지까지 구분하는지는 미검증(§12).
- **라벨(Label)** — 병의 표식. 촬영해 매칭·모델명 확인에 사용.
- **모델명 추론(LLM Inference)** — 라벨 사진을 LLM으로 분석해 와인 모델명을 자동 추정하는 것. 신규 등록의 기본값을 채우며, 직원이 항상 수정·확정한다.
- **매칭(Matching)** — 스캔한 바코드/라벨을 기존 와인 마스터와 대조해 후보를 찾는 과정.
- **입고 기록(Receiving Record)** — `와인 마스터 + 수량 + 입고일시 + 담당 직원 + 메모(선택)`. 하나의 입고 이벤트.
- **입고(Receiving)** — 와인 마스터를 확정하고 수량을 등록해 재고에 반영하는 행위.
- **리포트(Report)** — 입고 기록을 기간(일/주/월)으로 집계·시각화·내보내기한 결과.

## 4. Features

### 4.1 계정 및 인증

**Description:** 직원과 관리자가 각각 가입·로그인한다. 인증은 이메일+비밀번호. 관리자는 직원의 모든 기능에 더해 조회·리포트를 사용한다. `[ASSUMPTION: POC에서는 가입 승인 절차 없이 자유 가입, 이메일 인증 생략.]`

#### FR-1: 직원 가입/로그인
직원은 이메일+비밀번호로 가입하고 로그인할 수 있다. 실현 UJ-1, UJ-2.
**Consequences (testable):**
- 미인증 상태에서 스캔·입고 화면 접근 시 로그인 화면으로 유도된다.
- 동일 이메일 중복 가입은 거부된다.

#### FR-2: 관리자 가입/로그인 및 역할 구분
관리자는 이메일+비밀번호로 가입·로그인하며, 조회·리포트 기능에 접근할 수 있다. 실현 UJ-3.
**Consequences (testable):**
- 직원 계정은 관리자 전용 리포트 관리 기능에 접근할 수 없다. `[ASSUMPTION: 직원도 일/주/월 조회는 가능, 관리자는 추가로 엑셀·종합 리포트. 권한 경계 확정 필요.]`
- 역할은 계정에 귀속되어 로그인 시 해당 역할 화면이 노출된다.

### 4.2 입고 스캔 및 식별

**Description:** 직원이 입고 시 카메라로 바코드를 스캔하고 라벨 사진을 촬영하면, 시스템이 와인 마스터와 매칭해 결과를 보여 준다. 바코드로 1차 매칭하고, 라벨 사진으로 빈티지 등 바코드가 구분하지 못하는 차이를 보정한다. 매칭이 안 될 때는 **LLM이 라벨 사진을 분석해 모델명을 추론**해 신규 등록을 돕는다. 실현 UJ-1, UJ-2.

#### FR-3: 바코드 스캔
직원은 카메라로 병의 바코드를 스캔할 수 있다.
**Consequences (testable):**
- 스캔된 바코드 값으로 와인 마스터 매칭을 시도한다.
- 바코드 인식 실패 시 재시도 또는 라벨 사진 기반 흐름으로 넘어갈 수 있다.

#### FR-4: 라벨 사진 촬영
직원은 병의 라벨 사진을 촬영할 수 있고, 사진은 입고 기록/와인 마스터에 첨부·표시된다.
**Consequences (testable):**
- 촬영한 사진은 서버에 저장되고 매칭 결과 화면 및 이후 조회에서 표시된다.

#### FR-5: 와인 마스터 매칭 및 결과 표시
시스템은 스캔/촬영 입력을 와인 마스터와 매칭해 모델명·빈티지·대표사진을 화면에 표시한다. 실현 UJ-1.
**Consequences (testable):**
- 매칭 성공 시 모델명·빈티지·사진이 표시된다.
- 동일 바코드에 빈티지가 복수면 후보 목록을 제시하고 직원이 라벨 기준으로 선택한다. `[ASSUMPTION: 바코드가 빈티지를 구분하지 못할 수 있어, 후보 선택 UI로 처리 — 검증 대상.]`
- 매칭 실패 시 FR-6(신규 등록)로 유도한다.

#### FR-6: 신규 와인 등록 (LLM 라벨 모델명 추론)
매칭 실패 시 직원은 새 와인 마스터를 등록할 수 있으며, 시스템은 **LLM이 라벨 사진을 분석해 추론한 모델명**을 기본값으로 채워 준다. 직원은 이를 수정·확정한다. 실현 UJ-2.
**Consequences (testable):**
- "새로 추가하시겠습니까?" 확인 후 등록 폼이 뜨고, 모델명 필드에 **LLM이 라벨 사진에서 추론한 모델명**이 채워진다.
- 직원은 모델명·빈티지를 수정할 수 있고, 저장 시 새 와인 마스터가 생성된다.
- LLM 추론이 실패·저신뢰일 때 직원이 처음부터 수동 입력할 수 있는 폴백이 제공된다. (폴백 트리거: LLM이 결과를 반환하지 못함 / 신뢰도가 임계값 미만 / 직원이 "직접 입력"을 선택. `[ASSUMPTION: 신뢰도 임계값·판정 방식은 아키텍처 단계 확정.]`)
- `[ASSUMPTION: 모델명은 LLM 추론(자동), 빈티지는 직원 확인·입력. 바코드 값이 있으면 함께 저장.]`
- `[ASSUMPTION: 라벨 사진이 외부 LLM API로 전송될 수 있음 — 지연·비용·개인정보/데이터 전송 정책 확인 필요(§9, OQ-6).]`

### 4.3 입고 처리

**Description:** 품목이 확정되면 직원이 수량을 지정해 입고를 완료하고, 이후 수량을 수정할 수 있다. 입고 기록에는 특이사항(파손·명세서 불일치 등)을 적는 선택적 메모를 남길 수 있다. 실현 UJ-1, UJ-2.

#### FR-7: 수량 지정 및 입고 완료
직원은 확정된 와인 마스터에 수량을 지정하고 입고를 완료할 수 있다.
**Consequences (testable):**
- 입고완료 시 `와인 마스터 + 수량 + 입고일시 + 담당 직원`으로 입고 기록이 생성된다.
- 입고일시가 자동 기록된다.

#### FR-8: 입고 후 수량 수정
직원은 등록된 입고 기록의 수량을 수정할 수 있다.
**Consequences (testable):**
- 수량 수정 시 재고 집계에 반영되고, 수정 이력이 남는다. `[ASSUMPTION: 수정 이력 보존 — POC 필수 여부 확인.]`

#### FR-12: 입고 기록 메모(비고)
직원은 입고 기록에 자유 텍스트 메모(비고)를 선택적으로 남기고 수정할 수 있다. (idea.md 데이터모델의 "글" 필드를 충족)
**Consequences (testable):**
- 입고 등록·수정 시 메모를 입력·변경할 수 있다(빈 값 허용).
- 저장된 메모는 조회(FR-9) 목록의 해당 기록에서 확인된다.
- `[ASSUMPTION: 메모는 선택 입력, 자유 텍스트. 길이 제한·검색 필요 여부는 확정 필요.]`

### 4.4 조회 및 리포트

**Description:** 직원·관리자가 입고 데이터를 일/주/월 게시판 형태로 조회하고, 관리자는 종합 리포트(그래프)로 분석하고 엑셀로 내보낸다. 실현 UJ-3.

#### FR-9: 일/주/월 입고 조회 (게시판형)
사용자는 입고 기록을 일간/주간/월간 단위로 목록 조회할 수 있다.
**Consequences (testable):**
- 기간 선택 시 해당 기간의 입고 품목·수량·일시가 목록으로 표시된다.
- 사진이 함께 표시된다.

#### FR-10: 종합 리포트 (그래프 분석)
관리자는 입고 항목을 그래프 형태로 분석하는 종합 리포트를 볼 수 있다.
**Consequences (testable):**
- 기간별 입고량·품목 분포가 그래프로 시각화된다. `[ASSUMPTION: 막대/선 그래프로 기간별 입고 수량·상위 품목 표시. 구체 지표 확정 필요.]`

#### FR-11: 엑셀 다운로드
관리자는 조회·리포트 데이터를 엑셀 파일로 내려받을 수 있다.
**Consequences (testable):**
- 다운로드된 엑셀에 기간 내 입고 품목·수량·일시·담당자가 표 형태로 포함된다.

**Notes:**
- 그래프·종합 리포트(FR-10)는 회장/오너 보고·POC 데모의 임팩트 산출물이다 — 데이터 정확성뿐 아니라 "한눈에 보이는 그림"으로서의 완성도가 중요.
- `[NOTE FOR PM]` §4.1(직원+관리자)·§4.4(리포트 전체)·FR-6(LLM 추론)를 모두 POC에 포함하면 브리프의 POC 상업조건(1주·3페이지·약 200만원)을 초과한다. §6·§8 참조 — 일정/비용 재협의 대상.

### 4.5 초기 재고 등록 (초기 세팅)

**Description:** 입고가 발생할 때마다 신규 와인을 하나씩 추가하는 방법(FR-6) 외에, 사업체가 **이미 보유한 와인들을 초기에 일괄 등록**하는 초기 세팅 과정을 작업자가 애초에 가져갈 수 있다. 작업자가 보유 재고를 돌며 **바코드와 라벨 사진을 함께 촬영**해 와인 마스터를 미리 구축해 두면, 이후 실제 입고 시 바코드로 즉시 매칭되어 식별이 빨라진다. 이는 고객사 고유 와인 마스터 DB(자산·전환비용)의 초기 시드를 채우는 과정이기도 하다.

#### FR-13: 초기 재고 일괄 등록(바코드+라벨 초기 세팅)
작업자는 입고 흐름과 별개로, 보유 와인의 바코드와 라벨 사진을 함께 촬영해 와인 마스터를 사전 등록(초기 세팅)할 수 있다.
**Consequences (testable):**
- 작업자는 초기 세팅 모드에서 바코드 스캔 + 라벨 촬영을 연속으로 수행해 여러 와인 마스터를 등록할 수 있다.
- 초기 세팅으로 등록된 와인 마스터는 이후 입고(FR-3~FR-5) 시 바코드로 즉시 매칭된다.
- 신규 등록(FR-6)과 동일하게 LLM 모델명 추론 또는 직접입력을 사용할 수 있다.
- `[ASSUMPTION: 초기 세팅에서 보유 수량(재고 기준선)을 함께 입력할지 여부는 확정 필요 — 입력 시 재고 집계의 초기값이 된다.]`
- `[ASSUMPTION: 초기 세팅은 입고 기록(FR-7)을 생성하지 않는 '마스터 등록' 중심으로 본다 — 입고 이벤트와의 구분은 아키텍처/에픽 단계 확정.]`

## 5. Non-Goals (Explicit)

- **판매/유통 상태 관리(예약중·판매중·거래완료 등)** — 이 제품은 입고·재고에 집중한다.
- **품질 검수** — 검수는 "정확한 품목 + 수량"만. 와인 품질 판정은 하지 않는다.
- **회계/세무·전자세금계산서 연동** — 대상 아님.
- **소비자 대면 기능(주문·판매·마케팅)** — 내부 운영 도구다.
- **출고/판매 차감 재고** — v1은 입고 중심. 출고 관리는 후속.

## 6. MVP Scope

> 용어 주의: 브리프는 "POC(1주·200만원)"를 규정했으나, §4의 확정(관리자 역할·LLM 추론·리포트 전체)은 그 범위를 넘어선다. 아래 In Scope는 **확정된 기능 기준**이며, 실제 POC 계약 범위는 §8 Open Question OQ-1에서 재협의한다.

### 6.1 In Scope (MVP)
- FR-1 직원 가입/로그인, FR-2 관리자 가입/로그인·역할 구분
- FR-3 바코드 스캔, FR-4 라벨 사진 촬영
- FR-5 와인 마스터 매칭·결과 표시 (빈티지 후보 선택 포함)
- FR-6 신규 등록 + 라벨 LLM 모델명 추론
- FR-7 수량 지정·입고완료, FR-8 수량 수정, FR-12 입고 메모(선택)
- FR-9 일/주/월 조회, FR-10 종합 리포트(그래프), FR-11 엑셀 다운로드
- FR-13 초기 재고 일괄 등록(바코드+라벨 초기 세팅)

- 관리자 웹(Next.js) 화면 — v1은 모바일 우선. `[NOTE FOR PM: 웹 병행은 본 제품 단계에서 재검토.]`
- 오프라인 입고 — `[ASSUMPTION: POC는 온라인(서버 연결) 필수.]`

## 7. Success Metrics

**Primary**
- **SM-1**: **검수·입력 시간 50% 단축** — 명세서 대조 방식 대비 병당 평균 처리시간. **측정법:** 시연/파일럿 전 기존 수기 방식으로 N병 처리시간을 재어 기준선(baseline)을 잡고, 앱으로 동일 N병 처리시간을 비교. 목표 = 병당 평균 50% 이상 단축. 검증 FR-3~FR-8. `[ASSUMPTION: 50%는 목표치(고객은 "확 줄 것 같다"는 정성 기대) — 기준선 측정 후 고객과 확정.]`
- **SM-2**: **POC 핵심 시퀀스 완주** — 지정 와인 마스터 10종에 대해 스캔/촬영→식별→(신규등록)→수량→입고완료가 end-to-end 동작. 검증 FR-3~FR-8.
- **SM-3**: **바코드↔빈티지 판정 도출** — 바코드로 빈티지 구분 가능한지에 대한 명확한 결론(가능/불가능/부분). 검증 FR-5, FR-6.

**Secondary**
- **SM-4**: 오입력 발생률 감소 및 입고 데이터 정확도 향상. 검증 FR-5~FR-8.
- **SM-5**: 일/주/월 리포트·엑셀이 실제 보고에 활용됨. 검증 FR-9~FR-11.

**Counter-metrics (do not optimize)**
- **SM-C1**: **등록 정확도** — 속도(SM-1)만 좇다 잘못된 품목/수량 등록이 늘면 안 된다. SM-1을 상쇄.
- **SM-C2**: **LLM 추론 맹신 방지** — 자동 채움 편의(FR-6) 때문에 직원의 모델명·빈티지 확인을 건너뛰어 오등록이 늘면 안 된다.

## 8. Open Questions

1. **OQ-1 (범위·비용).** 확정된 기능(관리자 역할+LLM 추론+리포트 전체)은 브리프의 POC 조건(1주·3페이지·200만원)을 초과한다. POC 계약 범위를 어디까지로 할지, 초과분은 본 과제로 넘길지 **회장 보고 전 재협의 필요**. 관련 상업 레버(브리프 addendum): 회장 판단 시 POC 없이 본과제 직행 가능성, 포트폴리오 활용 동의 시 견적 할인 옵션 — 범위·비용 협상에 함께 고려.
2. **OQ-2 (바코드↔빈티지).** 바코드가 빈티지별로 다른지 미검증. 양측 확인 예정. 불가하면 식별을 라벨 사진 중심으로 재설계.
3. **OQ-3 (매칭 기준 데이터).** 고객사가 라벨↔코드(와인 마스터) 매칭 테이블을 제공/정리할 수 있는지 — 식별 방법론과 10종 선정에 직결.
4. **OQ-4 (권한 경계).** 직원과 관리자의 조회·리포트 권한 경계 확정.
5. **OQ-5 (LLM 추론 정확도).** 수입사별 라벨 다양성에서 LLM 모델명 추론이 실용적으로 동작하는지 — 실패 시 수동 입력 폴백.
6. **OQ-6 (LLM 운영: 지연·비용·데이터).** 제공자는 **Gemini/OpenAI(클라우드)** 로 정함(API 키 env 준비). 남은 확인: 둘 중 선택(정확도/비용/지연 POC 비교), 호출당 지연·비용, **라벨 사진 외부 전송에 대한 고객사 데이터 정책**(불가 시 온프렘/경량 모델 대안).

## 9. Cross-Cutting NFRs

- **성능/규모:** 하루 최대 ~100건 입고, 와인 마스터 ~1,000종 검색·매칭에서 스캔→결과 표시가 지연 없이 동작. `[ASSUMPTION: 매칭 결과 표시 목표 2초 이내. 목표치 확정 필요.]`
- **신뢰성:** 데이터는 **서버 중심**으로 저장(단순 로컬 앱 금지). 입고 기록은 유실되지 않아야 한다.
- **사용성:** 현장 한 손 조작·연속 스캔에 최적화. 최소 탭으로 입고 1건 완료.
- **보안:** 이메일+비밀번호 인증, 비밀번호 안전 저장. `[ASSUMPTION: POC 수준 기본 보안, 정식 보안 강화는 본 제품.]`
- **LLM 추론 응답성/비용:** 신규 등록 시 라벨 사진 → 모델명 추론이 현장 흐름을 끊지 않아야 함. `[ASSUMPTION: 추론 결과 표시 목표 3~5초 이내, 건당 비용·모델 선택은 아키텍처 단계 확정(OQ-6).]` 라벨 사진 외부 전송 시 데이터 정책 준수.

## 12. Risk and Mitigations

- **바코드↔빈티지 미검증 (최우선).** POC의 핵심 검증 목표. 완화: 후보 선택 UI + 라벨 사진 보정, 불가 시 라벨 중심 재설계. (OQ-2)
- **LLM 추론 정확도.** 라벨 다양성으로 모델명 추론이 흔들릴 수 있음. 완화: 항상 직원 수정 단계 유지, 저신뢰 시 수동 입력 폴백. (OQ-5, SM-C2)
- **LLM 운영(지연·비용·데이터 전송).** 제공자 Gemini/OpenAI(클라우드) 확정 → **라벨 사진 외부 전송이 확실**. 건당 지연·비용도 발생. 완화: 데이터 정책 합의 필수, 정책 불가 시 온프렘/경량 모델, 이미지 압축·프롬프트 최적화. 상세는 PRD `addendum.md`. (OQ-6)
- **매칭 기준 데이터 부재.** 고객사 자료 제공 여부가 관건. 완화: 10종 지정으로 POC 한정. (OQ-3)
- **범위 초과.** 확정 기능이 POC 예산·기간을 초과. 완화: OQ-1 재협의, MVP 단계 분리. (OQ-1)
