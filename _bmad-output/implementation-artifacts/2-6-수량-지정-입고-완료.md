# Story 2.6: 수량 지정 & 입고 완료 (FR7, UX-DR6·13)

Status: review

## Story

As a 직원,
I want 수량만 넣고 완료하면 입고가 확정되기를,
so that 한 병을 최소 탭으로 끝내고 다음 병으로 넘어간다.

## Acceptance Criteria

1. **`receiving_records` 스키마 + 마이그레이션**: `wine_vintage_id`(FK, indexed) · `quantity`(int, ≥1) · `received_at`(datetime, **서버 생성**) · `staff_id`(FK users, indexed) · `memo`(nullable str) · `deleted_at`(nullable datetime) · `created_at`. Alembic `0003_receiving.py`(`down_revision="0002_wine"`).
   - **`deleted_at`을 지금 만든다**(삭제 UI는 Epic 4). AR6는 soft-delete만 허용하며 하드삭제를 금지한다. 나중에 추가하면 그 사이에 작성된 모든 집계 쿼리가 필터를 빠뜨린 채 복사·전파된다.
2. **입고 생성 엔드포인트**: `POST /api/v1/receiving`(인증) — body `{wine_vintage_id, quantity, memo?}` → 201 + 생성된 레코드.
   - `received_at`은 **서버가 UTC로 생성**한다. 클라이언트 시각을 신뢰하지 않는다 — 공용 창고 단말 시계는 어긋나고, 이 기록은 5년 보존 원장(국세기본법 §85조의3)이다.
   - `staff_id`는 **`CurrentUser`에서만** 채운다. 요청 body의 staff_id는 받지도 읽지도 않는다 — 클라이언트가 담당자를 지정할 수 있으면 감사 추적이 무의미해진다.
   - `quantity < 1` → 422. 존재하지 않는 `wine_vintage_id` → 404.
3. **현재고 집계**: `crud.receiving.get_stock_map(session, vintage_ids) -> dict[UUID, int]` — **단일 GROUP BY 쿼리**(N+1 금지, NFR1 2초). `deleted_at IS NULL`을 **첫 쿼리부터** 필터한다.
   - 판매·출고는 범위 밖(Non-Goal)이므로 현재고 = 입고 수량 합계.
4. **스캔 응답에 현재고 노출**: `WineVintageRead`에 `stock: int` 추가 → 확인 카드의 현재고 배지(UX-DR5)가 실데이터로 표시된다. 기존 스캔 테스트는 깨지지 않아야 한다.
5. **QuantityStepper(UX-DR6)**: `[−] / 숫자(34pt) / [+]`, **최소 1**(1일 때 `[−]` 비활성), 숫자 탭 시 키패드 직접 입력. 터치 타깃 ≥48dp. 접근성 semantic 라벨(`수량 감소`·`수량`·`수량 증가`).
6. **하단 고정 [완료]**: 풀폭 58dp, 네이비. 누르면 `POST /receiving` → 성공 시 **그린 SnackBar + 체크 아이콘 + 햅틱**(UX-DR13) 후 스캔 화면으로 자동 복귀.
7. **⚠️ 완료 후 스캔 상태 완전 초기화**: `matchProvider` · `selectedCandidateProvider` **및 `scanControllerProvider.reset()`**을 모두 리셋한다.
   - **`reset()`을 빠뜨리면 같은 와인의 두 번째 병을 스캔할 수 없다.** `ScanController._lastAccepted`가 같은 코드 연속 인식을 디바운스로 무시하기 때문(`scan_controller.dart:20-25`). 같은 와인 여러 병 입고는 예외가 아니라 **가장 흔한 케이스**다.
8. **중복 제출 방지**: 제출 중에는 [완료]를 비활성화한다. 연타가 입고 레코드를 2건 만들면 재고가 조용히 부풀고, 직원은 알아챌 방법이 없다.
9. **실패 시 데이터 보존(NFR2)**: 네트워크·서버 오류 시 **스캔으로 복귀하지 않는다.** 입력한 수량과 선택한 후보를 유지한 채 인라인 오류 + 재시도 액션을 보여준다(차단 모달 금지).
10. **검증**: 백엔드 — 생성 201·서버 시각·staff 귀속·quantity 검증·404·미인증 401·현재고 집계(soft-delete 제외 포함). Flutter — 스테퍼 경계(1에서 `[−]` 비활성)·완료 성공 흐름·중복 제출 차단·실패 시 상태 보존·스캔 상태 리셋.

## Dev Notes

### 이번 스토리의 숨은 함정 3가지

**① 같은 와인 두 번째 병을 스캔할 수 없다.** `ScanController.onDetected`(`scan_controller.dart:20-25`)는 `_lastAccepted`와 같은 코드를 무시한다. 스캔 중 중복 인식을 막는 올바른 디바운스지만, **입고 완료는 "다음 병"의 시작**이므로 `reset()`으로 경계를 그어야 한다. 빠뜨리면 직원이 같은 와인 12병을 넣을 때 두 번째부터 카메라가 죽은 것처럼 보인다. 수량 스테퍼가 있으니 12병을 한 건으로 넣으면 되지 않냐고 할 수 있으나, 현장은 "찍고→수량→완료→다음 병" 리듬이고 병마다 확정하는 것이 UX 전제다.

**② [완료] 연타 = 재고 부풀림.** 네트워크가 느리면 직원은 반드시 다시 누른다. 서버에 멱등 키가 없으므로 클라이언트에서 제출 중 비활성이 유일한 방어선이다.

**③ soft-delete 필터를 나중에 붙이면 늦다.** `deleted_at IS NULL`이 첫 집계 쿼리에 없으면, 이후 재고·리포트·엑셀 쿼리가 그 쿼리를 복사하며 필터 없는 채로 번진다. Epic 4·5에서 되돌리려면 전수 감사가 필요하다.

### 서버가 시각과 담당자를 정한다

`received_at`을 클라이언트가 보내면 안 되는 이유는 편의가 아니라 신뢰성이다. 이 레코드는 5년 보존 원장이고(AR6·국세기본법 §85조의3), 공용 단말 시계는 어긋난다. 같은 이유로 `staff_id`는 JWT의 `CurrentUser`에서만 온다 — `ReceivingCreate` 스키마에 필드 자체를 두지 않으면 실수로 신뢰할 경로가 생기지 않는다.

### 현재고의 정의

판매·출고·예약은 명시적 Non-Goal이므로 **현재고 = `SUM(quantity) WHERE deleted_at IS NULL`**. 이 단순함은 지금은 옳지만 영구적이지 않다 — 출고가 생기면 이 정의가 바뀐다. `get_stock_map` 한 곳에만 두어 나중에 한 군데만 고치면 되게 한다. 화면에서 합산하지 말 것.

**N+1 금지**: 후보가 여러 개면 vintage마다 쿼리하고 싶어지지만, NFR1(스캔→표시 2초)이 걸려 있다. `IN` + `GROUP BY` 단일 쿼리로 맵을 만들어 주입한다.

- **디바운스 경계**: 완료 시 `scanController.reset()`을 호출한다. 없으면 `_lastAccepted` 때문에 **같은 와인 두 번째 병이 스캔되지 않는다** — 회귀 테스트로 고정했다(같은 코드가 완료 후 다시 `true`를 반환하는지).
- **중복 제출**: 300ms 지연 fake로 [완료] 연타 → 저장 1회만 발생함을 검증.
- **서버 권위**: `received_at`·`staff_id`를 body로 주입 시도하는 테스트를 넣어 무시됨을 확인(1999년 주입 → 현재 시각, 위조 staff_id → 토큰 사용자).
- **soft-delete**: `deleted_at` 컬럼과 집계 필터를 **동시에** 도입. 취소분 100병이 현재고에서 빠지는지 테스트로 고정.
