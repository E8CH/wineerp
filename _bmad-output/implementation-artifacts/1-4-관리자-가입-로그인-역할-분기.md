# Story 1.4: 관리자 가입/로그인 & 역할 분기 (FR2)

Status: review

## Story

As a 관리자,
I want 관리자로 로그인해 조회·리포트에 접근하기를,
so that 현장에서 입고 현황을 파악하고 보고할 수 있다.

## Acceptance Criteria

1. **역할 가드**: manager 전용 리소스에 접근하는 재사용 의존성(`require_manager`) 제공 — staff 토큰이면 403, manager면 통과.
2. **관리자 계정 생성**: `POST /api/v1/auth/managers`(email, password) → role=manager 계정 생성. **manager만 호출 가능**(staff 403). 응답 UserRead.
3. **역할은 계정 귀속**: 로그인 시 발급 토큰으로 `/auth/me`가 해당 role을 반환(로그인 화면 분기의 서버측 근거).
4. **부트스트랩 경로**: 최초 관리자 생성 수단 제공(자기참조 없이) — 시드/스크립트 또는 문서화된 절차. (OQ-4 권한 경계 확정 전까지 최소 구현.)

## Dev Notes

- **재사용**: `create_user(role=manager)`(1.2), `get_current_user`/`CurrentUser`(1.3) 존재 — 재구현 금지. `require_manager`는 `get_current_user` 위에 role 체크만 얹는다.
- **역할 = 단일 기준**: is_superuser 금지. manager ⊇ staff 권한(개념). 실제 리포트 라우트 가드는 Epic 5에서 `require_manager` 재사용. [Source: epics.md#Story 1.4 / architecture.md AR5]
- **부트스트랩 딜레마**: 관리자만 관리자를 만들 수 있으면 최초 1인은 별도 경로 필요 → `scripts/create_manager.py`(로컬/운영자 실행, DB 직접 crud)로 해결. 공개 API로 self-manager 승격은 열지 않는다(보안).
- **에러**: 권한 부족 403(`{"detail": ...}`), 중복 400. [Source: architecture.md#Implementation Patterns]
- **테스트**: 최초 manager는 override 세션에 `create_user(role=manager)`로 시드 후 로그인 토큰 사용.
