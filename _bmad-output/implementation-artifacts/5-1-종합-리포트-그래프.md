# Story 5.1: 종합 리포트 그래프 (FR10, UX-DR11)

Status: review

## Story

As a 관리자,
I want 기간별 입고량·품목 분포를 그래프로 보기를,
so that 회장/오너 보고에 한눈에 보이는 그림을 쓴다.

## 결정 두 가지

**① 차트 라이브러리를 추가하지 않는다.** UX-DR11의 요구는 "기간별 입고 막대(피크 골드) + 빈 상태 대응"이다. `fl_chart`는 자체 테마·축 체계를 들고 와 네이비 디자인 토큰과 싸우고, 막대 하나에는 과하다. 순수 위젯으로 그린다. 축·툴팁·다중 시리즈가 필요해지면 그때 재검토한다.

**② 버킷팅은 SQL이 아니라 Python에서 한다.** KST 기준 일별 묶음을 SQL로 하면 SQLite(`strftime`)와 PostgreSQL(`date_trunc … AT TIME ZONE`)이 갈리고, 4.1에서 겪은 시간대 함정을 방언마다 다시 밟는다. 월 최대 ~3,000행(하루 100건 × 31일, NFR1 규모)이라 애플리케이션에서 묶는 비용이 무의미하다. **경계 계산은 `core.timeframe.period_bounds`를 그대로 재사용**한다.

## Acceptance Criteria

1. **리포트 엔드포인트(manager 전용)**: `GET /api/v1/reports/receiving?period=week|month&anchor=YYYY-MM-DD` → `CurrentManager` 가드. staff는 **403**.
2. **기간은 주간/월간만**: 일간은 막대가 하나뿐이라 그래프의 의미가 없다. 주간=7개 일별 막대, 월간=해당 월 일수만큼.
3. **응답 구조**: `{buckets:[{date, quantity}], top_products:[{model_name, producer, quantity}], total_quantity, record_count, distinct_wines}`.
   - 버킷 날짜는 **KST 로컬 날짜**다(`YYYY-MM-DD`). 빈 날짜도 `quantity: 0`으로 채운다 — 빠지면 막대 간격이 거짓말을 한다.
   - `top_products`는 수량 상위 5종.
4. **soft-delete 제외**, 초기 세팅분(`source='initial_setup'`) **포함**. 재고에 반영되는 것은 리포트에도 반영되어야 일관된다.
5. **ReportBarChart(UX-DR11)**: 일별 막대, **최대값 막대는 골드**(`categoryStock`), 나머지는 네이비. 값 0인 날도 자리를 차지한다.
   - 색만으로 피크를 전달하지 않도록 최대값에 수치 라벨을 붙인다.
6. **KPI 표시**: 총 입고 수량·기록 건수·품목 수.
7. **빈 상태**: 기간에 데이터가 없으면 막대 대신 안내를 표시한다.
8. **staff는 리포트 탭에 접근할 수 없다**: 하단 탭에서 숨기거나 접근 시 안내. 서버 403과 이중 방어.
9. **검증**: 백엔드 — manager 200/staff 403·KST 일별 버킷·빈 날짜 0·상위 품목 정렬·soft-delete 제외·세팅분 포함·미인증 401. Flutter — 막대 렌더·피크 강조·빈 상태·KPI·staff 접근 차단.
10. **변이 검증**: manager 가드, KST 버킷팅, 빈 날짜 채우기를 각각 제거하면 테스트가 실패해야 한다. ([[test-mutation-verification]])

## Dev Notes

### 빈 날짜를 0으로 채우는 이유

입고가 없던 날을 빼면 막대가 붙어서 그려지고, 보는 사람은 "매일 들어왔다"고 읽는다. 그래프의 거짓말 중 가장 흔한 형태다. 기간의 모든 날짜를 채운다.

### 초기 세팅분을 포함하는 이유

`source`로 구분은 하되 리포트에서 **빼지는 않는다**. 재고에 반영되는 수량이 리포트에 없으면 "재고는 100인데 리포트 합계는 40"이 되어 관리자가 둘 중 무엇을 믿을지 모른다. 구분이 필요하면 후속에서 스택 막대로 나눌 수 있고, 데이터는 이미 `source`로 갈라져 있다.

### 이중 방어(서버 403 + UI 숨김)

UI만 숨기면 API는 그대로 열려 있고, 서버만 막으면 staff가 빈 화면과 오류를 본다. 둘 다 한다.
