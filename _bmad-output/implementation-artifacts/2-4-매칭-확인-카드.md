# Story 2.4: 매칭 & 확인 카드 (FR5-매칭, UX-DR5·9)

Status: review

## Story

As a 직원,
I want 스캔 즉시 모델명·빈티지 카드를 보기를,
so that 어떤 품목인지 곧바로 확신하고 수량 입력으로 넘어간다.

## Acceptance Criteria

1. **매칭 엔드포인트**: `POST /api/v1/scan`(인증, {code}) → 바코드 code로 WineProduct 후보 + 각 WineVintage 반환. 미매칭이면 빈 목록(프론트가 신규등록 유도).
2. **응답 구조**: products[{id·producer·model_name·region·country·grape·lwin7·vintages[{id·vintage·lwin11·representative_image_key}]}].
3. **확인 카드 위젯**: `ReceivingConfirmCard`(사진 자리·모델명 22pt·빈티지·좌측 CategoryBar). 단일 매칭 시 표시.
4. **미인증 401**.
5. **검증**: 백엔드 매칭 테스트(SQLite+시드), 카드 위젯 테스트.

## Dev Notes

- **재사용**: `crud.wine.find_products_by_barcode`·`get_vintages_for_product`(Story 2.1) — 재구현 금지. [Source: architecture.md#Data Flow]
- **분기**: 매칭 0 → 신규등록(3.2), 복수 빈티지 → 후보 선택(2.5). 2.4는 매칭 반환 + 단일 카드까지.
- **현재고**: receiving_records(2.6) 이전이라 stock은 이번 범위 밖 — 카드는 식별(모델명·빈티지) 중심, 현재고 배지는 2.6에서 연결.
- **위생**: 스캔 code는 그대로 매칭키(빈티지 파싱 없음). [Source: research]
- **프론트 한계**: 실제 스캔→API 호출 end-to-end는 Flutter 인증/ApiClient 필요(아직 미구축) → 이번엔 카드 위젯 + 리포지토리 로직만, 실연결은 후속(Flutter 인증 UI/클라이언트) 필요. 확인 카드는 샘플 데이터로 위젯 테스트.
