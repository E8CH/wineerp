# Story 4.3: 입고 메모 (FR12)

Status: review

## Story

As a 직원,
I want 입고 기록에 선택적 메모를 남기고 수정하기를,
so that 파손·명세서 불일치 등 특이사항을 기록한다.

## 이 스토리가 작은 이유

`memo` 컬럼은 2.6에서, 내역 표시는 4.1에서, 수정 시트는 4.2에서 이미 만들었다.
남은 것은 **입력 경로 두 곳**뿐이다: 입고 완료 시 남기기, 나중에 고치기.

## Acceptance Criteria

1. **입고 시 메모(선택)**: `ReceivingPanel`에 메모 입력을 접이식으로 둔다. 기본은 접힌 상태.
   - ⚠️ 항상 펼쳐 두면 "찍고→수량→완료" 3탭 리듬에 필드 하나가 끼어든다. 메모는 예외 상황용이므로 **기본 경로를 늦추면 안 된다**(NFR3).
   - `POST /receiving`은 이미 `memo`를 받는다 — 전송만 연결한다.
2. **메모 수정**: `PATCH /receiving/{id}`가 `memo`를 함께 받는다.
   - 빈 문자열은 **삭제**로 처리해 `null`로 저장한다. `""`와 `null`이 공존하면 "메모 있음" 판정이 두 갈래가 된다.
   - 메모만 바꾸는 것도 유효한 수정이다(수량 동일해도 저장된다).
3. **메모 변경도 이력에 남는다**: 수량과 함께 `receiving_amendments`에 기록한다(`before/after_quantity`는 동일값). 5년 보존 원장에서 **무엇이 바뀌었든** 추적 가능해야 한다.
4. **표시**: 4.1의 `HistoryRow`가 이미 메모를 보여준다 — 회귀만 확인한다.
5. **검증**: 백엔드 — 입고 시 메모 저장·메모만 수정·빈 문자열이 null로·이력 기록·길이 제한 422. Flutter — 접힘 기본·펼쳐 입력 후 전송·시트에서 메모 수정.
6. **변이 검증**: 빈 문자열→null 정규화, 메모 접힘 기본을 각각 제거하면 테스트가 실패해야 한다. ([[test-mutation-verification]])

## Dev Notes

### 메모를 기본 경로에서 치우는 이유

NFR3는 "최소 탭으로 입고 1건 완료"다. 메모는 파손·불일치 같은 **예외** 상황용이고 대부분의 병에는 필요 없다. 필드를 상시 노출하면 100병을 처리하는 직원이 100번 그것을 지나쳐야 한다. 접어두고 필요할 때만 펼친다.

### 빈 문자열을 null로 정규화하는 이유

`""`와 `null`이 둘 다 저장되면 "메모 있음" 판정이 `memo != null`인지 `memo.trim().isNotEmpty`인지 호출부마다 갈린다. 경계에서 한 번 정규화하면 이후 전부 `is None` 하나로 끝난다. 4.1의 `hasMemo`가 이미 trim 검사를 하고 있는데, 그건 방어일 뿐 계약이 아니었다.

- **메모만 바꿔도 이력에 남는다**: `update_quantity`를 `update_record`로 넓혀 수량·메모 어느 쪽이 바뀌어도 `receiving_amendments` 행이 생긴다. 5년 보존 원장에서 무엇이 바뀌었든 추적 가능해야 한다.

- **변이 검증 4건 전부 검출**: 빈 문자열 정규화 / `memo_provided` 무시(항상 덮어씀) / 메모 상시 노출 / 메모 미전송.
