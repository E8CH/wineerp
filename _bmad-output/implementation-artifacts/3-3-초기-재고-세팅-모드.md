# Story 3.3: 초기 재고 세팅 모드 (FR13, UX-DR10)

Status: review

## Story

As a 작업자,
I want 보유 재고를 연속으로 사전 등록하는 별도 모드를,
so that 초기 와인 마스터 DB를 구축해 이후 입고를 빠르게 만든다.

## 결정: PRD의 미확정 가정 두 건

PRD에 `[ASSUMPTION]`으로 열려 있던 항목을 여기서 닫는다.

**① 보유 수량(재고 기준선)을 함께 입력한다 — 선택 사항으로.**
세팅을 돌면서 병을 세는 것이 나중에 따로 세는 것보다 싸다. 다만 강제하면 "마스터만 빨리 등록하고 싶다"는 경우를 막으므로 선택으로 둔다.

**② 기준 재고는 `receiving_records`에 `source='initial_setup'`으로 기록한다.** 별도 테이블이 아니다.
- 재고 집계는 `crud.receiving.get_stock_map` **한 곳**에 있다. 기준 재고를 다른 테이블에 두면 재고·리포트·엑셀 쿼리가 전부 두 소스를 union해야 하고, 하나만 빠뜨려도 재고가 조용히 틀린다 — 2.6에서 `deleted_at`을 뒤늦게 붙이면 안 된다고 판단한 것과 같은 이유다.
- 에픽이 요구하는 "입고 이벤트와 **데이터로** 구분"은 `source` 컬럼이 충족한다. "**시각으로** 구분"은 골드 배너가 충족한다.
- 대안(별도 `stock_baseline` 테이블)은 개념적으로 깔끔해 보이지만, 세는 곳을 둘로 만드는 대가가 그 깔끔함보다 크다.

## Acceptance Criteria

1. **`source` 컬럼**: `receiving_records.source` enum(`receiving` | `initial_setup`), 기본 `receiving`. Alembic `0005_receiving_source.py`(`down_revision="0004_receiving_idempotency"`).
   - 기존 행은 전부 `receiving`으로 채워진다(NOT NULL + server_default).
2. **원자적 등록**: `POST /wines`에 선택 필드 `initial_quantity`(≥1) 추가. 있으면 마스터 생성과 기준 재고 기록이 **한 요청에서** 함께 생성된다.
   - ⚠️ 클라이언트가 `/wines` 후 `/receiving`을 따로 부르게 하면, 사이에서 실패했을 때 **수량 없는 마스터**가 남고 작업자는 알 수 없다.
3. **재고에 반영되되 구분 가능**: 기준 재고는 `get_stock_map` 합계에 포함되고, `source`로 입고분과 구분된다.
4. **SetupModeBanner(UX-DR10)**: 골드 배너 + **등록 카운터**(N종) + **명확한 나가기**. 입고 모드와 색으로 구분한다.
5. **연속 등록 리듬**: 등록 완료 시 `[등록하고 다음 병]`(골드)로 카운터가 오르고 폼이 비워지며 **모드는 유지**된다. 입고 흐름의 그린 SnackBar·스캔 복귀와 섞이지 않는다.
6. **세팅 모드에서는 입고가 일어나지 않는다**: 매칭된 바코드를 스캔하면 "이미 등록된 와인"으로 안내하고 수량 스테퍼·[완료]를 띄우지 않는다.
   - 세팅은 마스터 등록이 목적이다. 여기서 입고 카드가 뜨면 작업자가 초기 세팅 중에 입고를 만들어 재고가 이중 계상된다.
7. **미매칭은 곧바로 등록 폼**: 세팅 모드에서는 "새로 등록" 버튼을 한 번 더 누르지 않는다(리듬).
8. **검증**: 백엔드 — `source` 기본값·`initial_quantity` 원자적 생성·수량 없으면 레코드 미생성·재고 합계 반영·구분 조회. Flutter — 배너·카운터·나가기·연속 등록 시 모드 유지·매칭 시 입고 미노출.
9. **변이 검증**: `source` 기본값, 원자적 생성, "세팅 중 입고 미노출" 가드를 각각 제거하면 테스트가 실패해야 한다. ([[test-mutation-verification]])

## Dev Notes

### 세팅 중 입고를 막는 것이 이 스토리의 안전장치

작업자가 창고를 돌며 세팅하는 동안 이미 등록된 와인을 스캔하는 일은 흔하다. 그때 입고 확인 카드가 뜨면 **[완료]를 눌러 입고 기록이 생긴다** — 실제로는 아무것도 들어오지 않았는데 재고가 늘고, 그 행은 정상 입고와 구별되지 않는다(`received_at`도 서버 시각이라 자연스럽다). 모드가 다르면 화면도 달라야 한다.

### 진입점

`ScanScreen` AppBar 액션으로 진입한다. 카메라가 스캔 탭에 있으므로 모드 전환이 화면 이동을 동반하지 않는 것이 리듬에 맞다. 나가기는 배너에 둔다(UX 사양: "별도 모드 — 배너 진입 표시·명확한 나가기").

- **AC1~9 충족. Epic 3 구현 완료.** PRD에 열려 있던 `[ASSUMPTION]` 두 건(보유 수량 입력 여부 / 입고 이벤트와의 구분 방식)을 닫았다.

- **원자적 생성**: `/wines`가 `initial_quantity`를 받아 마스터와 기준 재고를 한 요청에서 만든다. 클라이언트가 `/wines` 후 `/receiving`을 따로 부르면 사이에서 실패했을 때 **수량 없는 마스터**가 남고 작업자는 알 수 없다.

- **연속 등록 리듬**: 미매칭이면 "새로 등록" 버튼을 한 번 더 누르지 않고 곧바로 폼이 뜬다. 등록 후 카운터가 오르고 폼이 비워지며 **모드는 유지**되고, `scanController.reset()`으로 같은 바코드도 다시 스캔된다.
- **모드 전환 시 상태 전면 정리**: 진행 중이던 스캔·선택·등록·수량을 모두 버린다. 모드가 다르면 화면도 달라야 하고, 넘어온 상태가 반대 모드에서 오작동하면 안 된다.
- **변이 검증 6건 전부 검출**: 세팅 중 입고 미노출 가드 / 보유 수량 전송 / 카운터 증가 / 나가기 시 카운터 초기화 / 원자적 기준재고 생성 / `source` 기본값.
- **⚠️ 변이가 드러낸 사실**: 모델의 `source` 기본값을 바꿔도 테스트가 통과했다. 테스트 결함이 아니라, **실제 경로에서 유효한 기본값은 `crud.create_record`의 시그니처 기본값**이고 모델 기본값은 그 뒤에 가려져 있기 때문이다(변이 6b로 확인). 3.1의 선조회 건과 같은 종류의 발견 — 보호가 실제로 어디에 있는지 알아야 나중에 엉뚱한 곳을 지우지 않는다.
