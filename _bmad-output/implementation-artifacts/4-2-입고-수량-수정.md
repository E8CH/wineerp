# Story 4.2: 입고 수량 수정 (FR8, AR6)

Status: review

## Story

As a 직원,
I want 잘못 넣은 입고 수량을 내역에서 수정하기를,
so that 실수를 되돌려 재고를 정확히 유지한다.

## Acceptance Criteria

1. **수정 이력 테이블**: `receiving_amendments`(`receiving_record_id` FK·`before_quantity`·`after_quantity`·`changed_by` FK users·`changed_at`·`reason` nullable). Alembic `0006_receiving_amendments.py`.
   - 에픽이 요구하는 "수정 이력이 보존된다"를 **행으로** 남긴다. 현재 수량만 덮어쓰면 무엇이 언제 왜 바뀌었는지 사라지고, 5년 보존 원장(AR6)의 의미가 없어진다.
2. **수정 엔드포인트**: `PATCH /api/v1/receiving/{id}`(인증) — `{quantity, reason?}` → 200 수정된 레코드.
   - 수량은 1~999(입고 생성과 동일 규칙). 0으로 만들려면 취소(AC4)를 쓴다.
   - 변경 없음(같은 수량)이면 이력 행을 만들지 않는다 — 잡음을 남기지 않는다.
   - 없는 id → 404, soft-delete된 레코드 → 404(이미 취소된 것은 수정 대상이 아니다).
3. **재고 즉시 반영**: 수정 후 `get_stock_map`이 새 수량을 반영한다(집계는 여전히 한 곳).
4. **🔴 취소는 manager 전용**: `DELETE /api/v1/receiving/{id}`는 `deleted_at`만 설정하는 soft-delete이며 **`CurrentManager` 가드**를 건다. staff는 403.
   - 코드리뷰 이월 항목이다. 취소는 5년 보존 원장에서 재고를 지우는 경로이고, 수정과 달리 **되돌릴 UI가 없다**. 수정(누구나)과 취소(관리자)의 권한을 다르게 두는 근거가 여기 있다.
   - 하드삭제 경로는 만들지 않는다(AR6).
5. **내역에서 수정 진입**: `HistoryRow` 탭 → 수정 시트. `QuantityStepper` 재사용, 사유(선택) 입력, [저장].
   - 저장 후 목록이 갱신된다. 취소 액션은 manager에게만 보인다.
6. **검증**: 백엔드 — 수정 200·이력 행 생성·동일 수량이면 이력 없음·재고 반영·404(없음/삭제됨)·422(범위)·취소 manager 200/staff 403·취소 후 재고 제외·미인증 401. Flutter — 시트 열림·수량 변경 저장·목록 갱신·staff에게 취소 미노출.
7. **변이 검증**: 이력 기록, manager 가드, soft-delete-only를 각각 제거하면 테스트가 실패해야 한다. ([[test-mutation-verification]])

## Dev Notes

### 왜 이력을 별도 행으로 남기는가

"수정 이력 보존"을 `updated_at` 하나로 때우면 **무엇이 얼마에서 얼마로 바뀌었는지**가 사라진다. 재고가 안 맞을 때 확인할 것이 없어지고, 국세기본법 §85조의3의 5년 보존이 요구하는 것도 최종 상태가 아니라 **장부와 증거서류**다. 행 하나 더 쓰는 비용이 훨씬 싸다.

### 수정과 취소의 권한이 다른 이유

수정은 "12를 11로" 같은 일상적 정정이고 잘못되면 다시 수정하면 된다. 취소는 재고에서 통째로 빼는 일이며 **이 스토리 범위에 복구 UI가 없다**. 되돌리기 비용이 다르면 권한도 달라야 한다. 코드리뷰가 이 비대칭을 지적했고 여기서 닫는다.

### soft-delete만 존재한다

`DELETE` 메서드를 쓰되 실제로는 `deleted_at`을 채운다. HTTP 동사와 저장 동작이 다른 것이 어색해 보일 수 있으나, 클라이언트에게 "취소"의 의미를 전달하는 데는 `DELETE`가 자연스럽고 AR6는 저장 계층의 규칙이다. **하드삭제 함수를 아예 만들지 않는 것**이 이 규칙을 지키는 방법이다.
