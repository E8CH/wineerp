# Story 2.5: 빈티지 후보 선택 (FR5-후보, UX-DR7)

Status: review

## Story

As a 직원,
I want 같은 바코드에 빈티지(또는 모델)가 여러 개일 때 라벨 기준으로 고르기를,
so that 바코드가 빈티지를 못 나눠도 정확히 확정할 수 있다.

## Acceptance Criteria

1. **후보 정렬 계약(백엔드)**: `/api/v1/scan` 응답의 `vintages`는 **최신 빈티지 우선, NV(null) 최후** 순으로 결정적 정렬된다. `get_vintages_for_product`가 `ORDER BY vintage DESC NULLS LAST`를 명시하며, PostgreSQL·SQLite 양쪽에서 동일 순서를 보장한다.
2. **분기 조건 = 평탄화 후보 수**: 스캔 결과를 `(product, vintage)` 쌍으로 평탄화한 개수 기준으로 분기한다.
   - 0개 → 미등록 안내(Epic 3 신규 등록 유도, 현행 유지)
   - 1개 → `ReceivingConfirmCard` 직행(현행 2.4 경로 유지)
   - **2개 이상 → `CandidateList` 표시**
   ⚠️ 제품이 1개여도 빈티지가 2개면 후보, 제품이 2개여도 각 1빈티지면 후보(총 2) — **제품 수 기준으로 분기하지 말 것**.
3. **CandidateList 위젯(UX-DR7)**: 썸네일 세로 카드 리스트. 각 행 = 썸네일 자리 + 모델명 + 빈티지 칩(`vintage?.toString() ?? 'NV'`) + 좌측 `CategoryBar(AppColors.categoryIdentity)`. 한 탭 선택 시 **네이비 하이라이트**(`AppColors.navy` 테두리 + `AppColors.container` 배경)로 선택 상태가 구분된다.
   - 제품이 복수면 각 행에 `producer`를 함께 표기해 동일 빈티지끼리 구분 가능해야 한다.
   - **차단 모달 금지**: 기존 `ScanScreen`의 하단 슬롯(`Stack`+`Positioned`, `scan_screen.dart:37-42`)에 `ReceivingConfirmCard`를 대체해 인라인 렌더. 별도 화면 push·Dialog·차단 모달 금지 — "연속 스캔마다 확인 모달(리듬 파괴)"은 명시된 안티패턴.
4. **선택 → 확인 카드 연결**: 후보 한 탭 선택 시 선택된 `(product, vintage)`로 `ReceivingConfirmCard`가 이어진다. 선택 이전에는 확인 카드가 보이지 않는다.
   - ⚠️ **"가장 최근 빈티지" 등 기본 선택 금지.** 후보가 2개 이상이면 사람이 반드시 한 번 탭한다.
5. **"목록에 없음" 탈출구**: 리스트 하단에 `찾는 빈티지가 없어요` 행을 제공하고, 탭 시 신규 등록(Epic 3) 유도 안내로 전환된다. 3.2 미구현 상태이므로 이번 스토리는 **안내 문구까지**(미매칭 케이스와 동일한 처리).
6. **재선택**: 후보 선택 후에도 `다시 선택` 액션으로 CandidateList로 되돌아갈 수 있다(UX-DR5 "취소·재선택", UX 사양 "오선택 즉시 취소/재선택").
7. **접근성·레이아웃(UX-DR15)**:
   - 각 후보 행 터치 타깃 **≥48dp**, 행 간격 12~14dp(오조작 방지), 카드 라운드 16dp
   - `Semantics` 라벨(예: `"샤토 마고 2018년, 후보 1/3, 선택됨"`)
   - 선택 상태를 **색만으로 전달 금지** — 체크 아이콘 + `선택됨` 텍스트 병기 (네이비 하이라이트 단독은 사양 자체 위반)
   - 폴드 대응: 입고 루프는 **단일 컬럼 중앙 정렬 유지, 2-pane 금지**. 최소폭 280dp에서 레이아웃 무결, 가로 스크롤 금지, 모델명 최소 20pt 유지
8. **폴드 전환 상태 보존(UX-DR14)**: 선택 상태는 **Riverpod 프로바이더에 보관**(위젯 로컬 `State` 금지). 접기/펴기는 Activity 구성 변경으로 위젯 트리를 재빌드하므로, 로컬 state면 스캔한 바코드·후보·선택이 사라진다.
9. **검증**: 백엔드 정렬 테스트(빈티지 혼합 + NV 포함), CandidateList 위젯 테스트(렌더·선택·하이라이트·탈출구 행), 분기 로직 테스트(0/1/2+).

## Dev Notes

### 이 스토리가 고치는 실제 결함

`app/lib/features/scan/scan_screen.dart:84-86`:

```dart
final product = result.products.first;
final vintage = product.vintages.isNotEmpty ? product.vintages.first.vintage : null;
```

`.first`가 **나머지 후보를 조용히 버린다.** 시드의 `SHARED-8801234567890`(Monte Bello / Geyserville 2제품)이나 `3760000000015`(샤토 마고 2015·2018)를 스캔하면 지금은 임의의 하나가 확정된다. 사용자에게는 정상 동작처럼 보이지만 **틀린 재고 단위로 입고된다.**

### 왜 정렬이 백엔드 계약인가

`get_vintages_for_product`(`backend/app/crud/wine.py:84-91`)에 현재 `ORDER BY`가 없어 순서가 비결정적이다. 기존 테스트가 `sorted(v["vintage"] for v in p["vintages"])`(`test_scan.py:58`)로 우회 중 — 즉 순서를 검증하지 않는다. 후보 목록에서 **순서는 곧 UX**(첫 항목이 가장 눌릴 확률이 높음)이므로 계약으로 승격한다. 프론트 정렬로 처리하면 2.6·4.1·재고 화면이 각자 정렬을 재구현하게 된다.

**NULLS LAST를 명시하는 이유**: PostgreSQL은 `DESC`에서 NULL이 **먼저**, SQLite는 **나중에** 온다. 테스트(SQLite)는 통과하고 운영(PostgreSQL)에서 NV가 목록 맨 위로 올라오는 전형적 함정. `nullslast()`가 양쪽을 통일한다.

### NV는 "인식 실패"가 아니다

`vintage` nullable은 의도적 설계(AR2). 빈티지 표기는 법적으로 **선택**이며 샴페인 85~95%·셰리 ~98%가 NV다. UI에서 NV를 오류·경고·회색 처리하지 말 것 — `ReceivingConfirmCard:27`의 `vintage?.toString() ?? 'NV'` 패턴을 그대로 따른다. 정렬상 마지막인 것은 숫자 축에 자리가 없어서일 뿐 열등해서가 아니다.

### "목록에 없음" 탈출구가 필요한 이유

업계 표준 결론은 **"사람에게 묻는다"**(InVintory는 빈티지 자동 인식을 시도조차 않고 항상 피커, CellarTracker는 스캔→후보→사용자 확인). 목록이 곧 진실이라고 가정하면, 실제 병의 빈티지가 마스터에 없을 때 직원은 **가장 가까운 틀린 항목을 고르고 넘어간다.** 빈티지가 가격결정·재고 단위인 이 제품에서 그건 침묵하는 데이터 오염이며, 제품의 존재 이유를 정면으로 깬다. UX-DR7 원문에는 없으나 필수로 포함한다.

### 피커는 예외 경로가 아니라 정상 경로다

아키텍처 강제 규칙: *"빈티지는 WineVintage에만, 바코드에 인코딩 금지. `vintage` nullable, **NV를 에러 취급 금지**"*, 안티패턴 목록에 *"바코드 연도 파싱"* 명시. 스캔 흐름의 정의 자체가 *"바코드→WineProduct 후보→**사용자가 라벨로 WineVintage 확정**(InVintory 항상-피커 = 업계 표준 해법)"*이다. 자동 추론·기본 선택으로 탭을 줄이려는 최적화는 이 스토리의 목적을 무효화한다.

- **빈티지 0개 제품**은 후보에 남기되 `isSelectable=false`(`빈티지 미등록`). 목록에서 빼면 스캔 결과가 조용히 사라지고, 선택 가능하게 두면 `wine_vintage_id` 없이 입고를 시도하게 된다.
- **접근성**: 선택 표시를 네이비 하이라이트 단독으로 두지 않고 체크 아이콘 + `선택됨` 텍스트 + `Semantics(selected:)` 병기. 사양의 "색만으로 상태 전달 금지"를 하이라이트 요구사항보다 우선했다.
- **폴드 대응**: 선택 상태를 `selectedCandidateProvider`에 보관(위젯 로컬 state 금지). 280dp 오버플로 테스트 포함.
- **의도적 미실행**: `ReceivingConfirmCard.onConfirm`은 배선하지 않았다. 이동 대상(QuantityStepper)이 2.6에 있어 지금 연결하면 아무 데도 가지 않는 버튼이 된다.
- **한계(변함없음)**: 실기 카메라·실서버 호출·Alembic 실행은 기기/배포 필요 → SQLite·위젯 테스트로 검증. 썸네일은 `representative_image_key`→URL 해석기가 없어 아이콘 placeholder 유지.
- **후속 인계**: 탈출구("찾는 빈티지가 없어요")는 현재 미등록 안내로만 전환된다 — Story 3.2에서 실제 신규 등록/빈티지 추가 흐름에 연결할 것. 현재고 배지는 2.6.
