# Story 4.1: 입고 내역 조회 (FR9, UX-DR12)

Status: review

## Story

As a 직원/관리자,
I want 입고 기록을 일/주/월로 조회하기를,
so that 특정 기간에 무엇이 얼마나 들어왔는지 확인한다.

## Acceptance Criteria

1. **조회 엔드포인트**: `GET /api/v1/receiving?period=day|week|month&anchor=YYYY-MM-DD`(인증) → `{data, count}`.
   - `anchor` 생략 시 오늘. `period` 생략 시 `day`.
   - 정렬은 **최신 입고 우선**(`received_at DESC`), 동점은 `id`로 고정.
2. **🔴 기간 경계는 KST 기준으로 계산한다**: `received_at`은 UTC로 저장되지만 "일간/주간/월간"은 한국 달력이다.
   - UTC로 경계를 자르면 **오전 9시(KST) 이전 입고가 전부 전날로 분류**된다. 창고 작업이 오전에 몰리므로 매일 상당수가 어긋난다.
   - 주 시작은 **월요일**(ISO). 월은 해당 월 1일~말일.
   - `settings.TIMEZONE`(기본 `Asia/Seoul`)로 두어 나중에 바꿀 수 있게 한다.
3. **soft-delete 제외**: `deleted_at IS NULL`만 반환한다(AR6).
4. **행 데이터(UX-DR12)**: `id·모델명·생산자·빈티지·수량·입고일시·담당자·메모·대표사진 key·source`.
   - `source`를 포함해 초기 세팅분(3.3)을 화면에서 구분할 수 있게 한다.
   - 조인은 **한 번의 쿼리**로 — 행마다 와인·담당자를 따로 조회하면 N+1이 된다.
5. **HistoryRow 위젯**: 사진 자리·모델명·시간·담당·수량 강조·메모 있으면 표시. 좌측 `CategoryBar`(입고=골드 / 초기세팅=네이비로 구분).
6. **로딩·빈 상태**: 리스트 로딩은 **스켈레톤**(빈 화면 금지), 기록이 없으면 안내 문구 + 스캔으로 유도.
7. **검증**: 백엔드 — 기간 필터·**KST 경계**(UTC 자정 전후 레코드가 올바른 날에 속하는지)·soft-delete 제외·정렬·조인 1회·미인증 401. Flutter — 세그먼트 전환·스켈레톤·빈 상태·메모 표시·source 구분.
8. **변이 검증**: KST 변환, soft-delete 필터, 정렬을 각각 제거하면 테스트가 실패해야 한다. ([[test-mutation-verification]])

## Dev Notes

### KST 경계가 이 스토리의 유일한 어려운 부분

나머지는 평범한 목록 조회다. 시간대만 틀리면 **화면은 정상으로 보이면서 매일 오전 입고가 전날에 잡힌다** — 사용자가 "어제 것이 왜 여기 있지"라고 물을 때까지 아무도 모른다. 관리자 리포트(Epic 5)가 같은 경계를 쓰게 되므로 여기서 한 번 제대로 만들어 재사용한다.

**테스트는 UTC 자정을 사이에 둔 두 레코드로 고정한다.** KST 변환이 없으면 반드시 실패하도록.

### 조회 권한

현재 스키마에 매장·테넌트 개념이 없어 인증된 사용자는 전체 내역을 본다. 관리자는 당연히 전체를 봐야 하고 직원도 자기 창고 전체를 보는 것이 자연스러우므로 지금은 문제가 아니다. 다만 **다점포로 확장하면 이 쿼리가 첫 번째 감사 대상**이라는 점을 남겨둔다.

### `source`를 내려보내는 이유

3.3에서 기준 재고와 입고를 같은 테이블에 두되 `source`로 구분하기로 했다. 내역 화면이 이를 구분하지 못하면 작업자가 "초기 세팅으로 넣은 10병"을 "오늘 입고된 10병"으로 읽는다.
