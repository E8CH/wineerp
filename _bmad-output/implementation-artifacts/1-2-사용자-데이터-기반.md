# Story 1.2: 사용자 데이터 기반 (User + role)

Status: review

## Story

As a 개발팀,
I want User 엔티티와 역할(staff/manager) 스키마·마이그레이션을,
so that 인증과 역할 분기의 데이터 토대가 마련된다.

## Acceptance Criteria

1. **User 모델**: `users` 테이블 = `id`(UUID PK), `email`(unique, not null), `hashed_password`, `role`(enum staff|manager, 기본 staff), `is_active`(bool 기본 true), `created_at`. SQLModel로 정의.
2. **역할 = 권한 단일 기준**: `role` enum이 staff|manager만 허용. 템플릿의 `is_superuser` 대신 `role`이 권한 판정 기준(후속 인증 스토리에서 사용).
3. **마이그레이션**: Alembic 마이그레이션으로 `users` 테이블 생성. `email`에 unique 제약. autogenerate가 User 모델을 인식(env.py import).
4. **비밀번호 해시 유틸**: `hashed_password`는 평문 저장 금지 — bcrypt 해시 함수(`get_password_hash`/`verify_password`) 제공(NFR4).
5. **DATABASE_URL 스킴 정규화**: `postgresql://`(Railway 제공) → `postgresql+psycopg://`로 정규화해 psycopg3 드라이버와 호환.

## Dev Notes

- **스택**: SQLModel(테이블), passlib[bcrypt](해시), Alembic(마이그레이션). 이미 pyproject에 포함.
- **명명 규약**(architecture): 테이블 snake_case 복수 `users`, 컬럼 snake_case, PK=UUID `id`, 타임스탬프 `created_at`. [Source: architecture.md#Implementation Patterns]
- **역할**: `role` enum(staff|manager) = 권한 단일 기준. is_superuser 사용 금지(anti-pattern). [Source: epics.md#Story 1.2 / architecture.md AR5]
- **DB 지연 초기화**: `core/db.py`는 DATABASE_URL 미설정 시 엔진 생성 안 함(health 무의존 유지). 테스트는 SQLite in-memory 엔진 주입.
- **마이그레이션 방침**: 테이블은 필요 시점 생성 — 이 스토리에서 `users`만. 와인 테이블은 Story 2.1.
- **테스트 DB**: Postgres 없이 SQLite(`sqlite://`)로 모델·crud 검증(SQLModel은 DB 비의존). psycopg는 Postgres 전용이라 테스트엔 불필요.

### Completion Notes List

- AC1~5 전부 충족·로컬 검증(SQLite in-memory 테스트).
- **편차: passlib → bcrypt 직접 사용.** passlib 1.7.4가 bcrypt 4.x와 비호환("error reading bcrypt version" + 72바이트 ValueError). passlib은 미유지보수라 `bcrypt` 라이브러리를 직접 사용하고 72바이트 절단을 명시 처리. pyproject 의존성도 `passlib[bcrypt]` → `bcrypt`로 교체.
- 마이그레이션(`0001_users.py`)은 Postgres 대상 — 로컬 Postgres 부재로 실행 검증은 배포 시(Railway). 모델·CRUD는 SQLite로 검증. role은 sa.Enum(name="userrole"), 값=이름("staff"/"manager") 일치.
- `UserRole`을 `StrEnum`(3.11+)으로.
