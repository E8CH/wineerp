# Story 6.1: 라벨 사진 저장·조회 (FR4 완결)

Status: ready-for-dev

> 에픽 미명시 — 실기 검증(2026-07-19)에서 드러난 FR4 미완결분. 사진을 찍어도 **어디서도 다시 볼 수 없었다.**

## Story

As a 직원,
I want 한 번 찍은 라벨 사진이 이후 스캔·목록에서 그대로 보이기를,
so that 글자를 읽지 않고 사진으로 병을 알아본다.

## 문제

FR4는 "사진은 서버 저장 후 매칭 결과·입고 기록·와인 마스터에 첨부·표시된다"고 요구한다. 지금은 **저장도 조회도 불완전**하다.

1. **저장이 휘발성**: R2 자격증명이 없어 `StoragePort`가 컨테이너 로컬 디스크로 폴백한다. Railway는 배포마다 컨테이너를 교체하므로 **업로드된 사진이 전부 사라지고** DB의 `representative_image_key`만 남아 깨진 참조가 된다.
2. **조회 경로가 아예 없다**: `/images`에 POST만 있고 GET이 없다. 확인 카드·후보 목록·내역 행이 와인잔 아이콘을 보여주는 것은 위젯 문제가 아니라 **가져올 방법이 없어서**다.

## 결정: R2 대신 `DbStorageAdapter`

아키텍처 안티패턴에 "이미지 base64 DB 저장"이 있으나, 그것은 일반론이고 이 프로젝트의 규모는 다르다.

- 사진은 **와인 1종당 1장**(`WineVintage.representative_image_key`)이지 입고마다가 아니다. 마스터 1,000종 × 압축 후 ~300KB ≈ **300MB** — Postgres가 감당하는 크기다.
- **base64가 아니라 `bytea`**로 넣는다. 안티패턴이 경계한 33% 용량 낭비가 없다.
- 벤더·자격증명·실패 모드가 늘지 않고, 백업에 사진이 자동 포함된다(5년 보존 원장에 유리).
- **`StoragePort` 뒤에 두므로 R2로 옮기는 비용은 환경변수 하나**다. Ports & Adapters를 채택한 이유가 이것이다.

⚠️ **이 결정이 무너지는 조건**: 입고 **건별** 사진을 저장하기 시작하면(하루 100건 × 5년 = 18만 장) DB는 부적절해진다. 그때는 R2로 전환한다.

## Acceptance Criteria

1. **`stored_images` 테이블**: `key`(PK) · `content_type` · `data`(LargeBinary) · `size` · `created_at`. Alembic `0008_stored_images.py`.
2. **`DbStorageAdapter`**: `put_object`/`get_object` 구현. 같은 key 재업로드는 덮어쓴다.
3. **어댑터 선택 순서**: R2 자격증명 있으면 R2 → 없으면 **DB**. 로컬 파일 어댑터는 테스트·개발 전용으로 남긴다(운영 기본값에서 제외 — 휘발성이 문제였다).
4. **`GET /api/v1/images/{key}`(인증)**: 바이트를 `content_type`으로 반환. 없으면 404.
   - ⚠️ **인증 필수**. 없으면 key를 아는 누구나 고객사 재고 사진을 본다.
   - key에 `/`가 포함되므로 path 파라미터로 받되, `..`·절대경로를 거절한다.
   - 캐시 헤더를 준다(`Cache-Control`, `ETag`) — 같은 병을 반복 스캔한다.
5. **앱에서 실제 사진 표시**: 확인 카드 · 후보 목록 · 내역 행.
   - 인증 헤더가 필요하므로 `Image.network`를 쓸 수 없다. dio로 바이트를 받아 표시한다.
6. **기기 캐시**: 한 번 받은 사진은 메모리·디스크에 두고 재요청하지 않는다. 창고에서 같은 와인을 반복 스캔하는데 매번 받으면 느리고 데이터를 쓴다.
7. **검증**: 백엔드 — 업로드→조회 왕복 · 미인증 401 · 없는 key 404 · 경로 이탈 거절 · 어댑터 선택 · 덮어쓰기. Flutter — 사진 있으면 표시·없으면 placeholder · 캐시 적중 시 재요청 없음.
8. **변이 검증**: 인증 가드, 경로 이탈 방어, 캐시를 각각 제거하면 실패해야 한다. ([[test-mutation-verification]])

## Dev Notes

### 왜 `Image.network`를 못 쓰는가

조회에 인증이 걸리므로 URL만으로는 못 가져온다. 인증을 빼면 key를 아는 누구나 고객사 재고 사진을 볼 수 있고, key는 응답에 담겨 나간다. dio로 받아 `Image.memory`로 그린다.

### 캐시가 선택이 아닌 이유

창고 순회는 같은 와인을 반복 스캔하는 작업이다. 캐시가 없으면 스캔할 때마다 수백 KB를 받는다 — 현장 리듬(NFR3)과 데이터 요금 양쪽에 영향이 있다.
