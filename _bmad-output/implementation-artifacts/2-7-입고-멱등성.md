# Story 2.7: 입고 멱등성 (코드리뷰 이월, NFR2)

Status: review

> 에픽 미명시 — Story 2.6 코드리뷰(2026-07-19)에서 이월된 High 결함. 원장 정확성 직결.

## Story

As a 직원,
I want 응답이 유실돼 다시 [완료]를 눌러도 입고가 한 번만 기록되기를,
so that 창고 와이파이가 끊겨도 재고가 조용히 부풀지 않는다.

## 문제

`POST /receiving`에 멱등 키가 없다. 요청이 서버에 도달해 **커밋된 뒤** 응답이 유실되면(타임아웃·소켓 절단·프록시 오류) 클라이언트는 실패로 판단하고, 직원은 [완료]를 다시 누른다 → 입고 레코드 2건.

`received_at`이 서버 생성이라 **두 행 모두 정상으로 보이고**, 사후 감사로도 구분되지 않는다. 2.6의 "제출 중 [완료] 비활성"은 **동시 제출만** 막지 순차 재시도는 막지 못한다. AR6상 정정 경로는 soft-delete뿐인데, 그러려면 **누군가 먼저 알아채야 한다.**

## Acceptance Criteria

1. **스키마**: `receiving_records.idempotency_key`(UUID, nullable, **unique**) + Alembic `0004_receiving_idempotency.py`(`down_revision="0003_receiving"`). nullable인 이유는 키 없는 호출(수동 스크립트·향후 배치)을 막지 않기 위해서다. PostgreSQL·SQLite 모두 unique 인덱스에서 NULL은 서로 충돌하지 않는다.
2. **재생 시 새 레코드를 만들지 않는다**: 같은 `idempotency_key`로 다시 POST하면 **기존 레코드를 그대로 반환**하고 `200`(생성 아님)을 준다. 최초 생성은 `201` 유지.
3. **경합 안전**: 동일 키 두 요청이 동시에 들어와 unique 제약에 걸리면, `IntegrityError`를 잡아 기존 레코드를 재조회해 반환한다. 500으로 새지 않는다.
4. **키 없는 요청은 현행대로**: `idempotency_key` 미포함 시 항상 새 레코드 생성(201). 기존 동작·테스트 불변.
5. **클라이언트가 병 단위로 키를 발급**: `ReceivingState`가 키를 보유하고, **재시도는 같은 키를 재사용**한다. 입고 성공 후 상태가 리셋되면 **새 키**가 발급된다(다음 병은 별개 입고).
   - 새 의존성 추가 금지 — `Random.secure()` 기반 UUID v4를 직접 생성한다.
6. **검증**: 백엔드 — 같은 키 2회 → 레코드 1건·두 번째는 200·동일 id 반환 / 키 없으면 2건 / 다른 키면 2건 / 경합 시 500 아님. Flutter — 실패 후 재시도가 **같은 키**를 보내고, 성공 후에는 **다른 키**가 발급된다.
7. **변이 검증**: 멱등 처리를 제거하면 백엔드 테스트가 실패해야 한다. 키 재사용 로직을 제거하면 Flutter 테스트가 실패해야 한다. (정책: [[test-mutation-verification]])

## Dev Notes

### 왜 200/201을 구분하는가

재생은 "생성됐다"가 아니다. 201로 거짓말하면 클라이언트·로그·향후 웹훅이 새 입고가 생겼다고 오해한다. 200 + 동일 레코드가 정직하다. 클라이언트 입장에선 둘 다 성공이므로 UX는 동일하다.

### 왜 서버 측이어야 하는가

2.6은 클라이언트 가드만 두고 커밋 메시지에 "중복을 막는다"고 썼는데, 코드리뷰가 그 주장이 **순차 재시도에 대해 거짓**임을 밝혔다. 네트워크 실패는 정의상 클라이언트가 결과를 모르는 상태이므로, **결과를 아는 쪽(서버)만이 중복을 판정할 수 있다.**

### 키의 수명

레코드에 영구 보존한다(원장이므로 어차피 5년). 별도 TTL·정리 배치를 두지 않는다 — 상태를 하나 더 만들면 그것이 또 틀릴 수 있다.

### Completion Notes List

- **AC1~7 충족.** 응답 유실 후 재시도가 더 이상 입고를 2건 만들지 않는다.
- **변이 검증 4건 수행**([[test-mutation-verification]] 정책):
  - A. 모델의 `unique=True` 제거 → **실패** ✅
  - B. 라우트의 `IntegrityError` 폴백 제거 → **실패** ✅
  - C. 재시도 시 새 키 발급하도록 변경 → **실패** ✅
  - D. 성공 후 새 키 미발급 → **실패** ✅
- **⚠️ 변이가 드러낸 설계 사실**: 선조회(`find_by_idempotency_key`)를 제거해도 테스트가 전부 통과했다. 처음엔 테스트 결함으로 의심했으나, 실제로는 **선조회가 최적화이고 진짜 보호는 unique 제약 + IntegrityError 폴백**이기 때문이다. 변이 A/B가 이를 확인했다. 보호의 소재를 잘못 알고 있으면 나중에 "중복 쿼리"라며 제약을 지우는 일이 생긴다.
- **경합 테스트를 라우트를 거치도록 재작성**: 최초 버전은 세션에 직접 INSERT해 제약만 확인했고 라우트의 폴백을 한 번도 실행하지 않았다(변이 B 미검출). `find_by_idempotency_key`의 **첫 호출만** 못 찾은 것처럼 monkeypatch해 실제 경합 창을 재현하도록 고쳤다.
- **200 vs 201**: 재생은 생성이 아니므로 200으로 답한다. 201로 답하면 클라이언트·로그·향후 웹훅이 새 입고가 생겼다고 오해한다.
- **`Random.secure()` 사용 이유**: 기본 `Random()`은 시드가 예측 가능해 같은 순간 시작한 두 단말이 같은 키를 낼 수 있고, 그러면 **서로의 입고를 삼킨다**(멱등성이 정반대로 작동).
- **soft-delete된 레코드도 키로 조회된다**: 취소된 입고를 재시도가 되살리면 안 되고, 그건 새 입고도 아니다.
- **한계**: 마이그레이션 0004는 작성만 되고 실행은 배포 이후(로컬은 SQLite `create_all`). 실제 동시 요청 경합은 monkeypatch로 근사했다 — 진짜 병렬 워커 검증은 배포 후 가능.
